# Doss 招聘链路重构实施方案（采集 + 候选人 + 面试 + 决策）

文档日期：2026-02-26  
适用范围：`apps/desktop`、`apps/desktop/src-tauri`、`packages/shared`、`packages/crawler-sidecar`

## 1. 目标摘要

本次重构目标：
1. 将采集链路重构为“多站采集 → 入待定候选人 → 去重归并 → 可选自动同步候选人”。
2. 将 AI 评估结果结构化输出，支持前端直接解析展示。
3. 重构候选人、面试、决策三大页面为分页查询与流程闭环。
4. 保持现有 `Vue3 + Pinia + Tauri + SQLite` 架构，优先复用现有 UI 组件。

确认结论（已锁定）：
1. “又想”字段按 `email` 处理。
2. AI 评估结果存储在 `screening_results` 扩展字段中。
3. 新增 `pending_candidates` 表作为待定候选人主表。
4. 调度支持 `单次 + 每日固定时刻 + 每月固定日时刻`。
5. 去重规则：姓名全等 + 年龄全等（任一为空不比）+ 地址全等（任一为空不比）。
6. 决策动作统一写入 `hiring_decisions`，并同步候选人阶段。
7. 列表统一后端分页查询。

## 3. AI 评估结构化输出规范

写入 `screening_results.structured_result_json`，结构约定：

```json
{
  "weights": {
    "t0": {
      "score": 4.2,
      "rule": "<3不匹配，3-4建议，>=4匹配",
      "matched": true,
      "details": ["..."]
    },
    "t1": {
      "template": "默认筛选模板",
      "items": [
        {"key": "goal_orientation", "label": "目标导向", "weight": 30, "score": 78, "reason": "..."}
      ]
    },
    "t2": {
      "bonus": 6,
      "items": ["..."]
    }
  },
  "risk_alerts": ["..."],
  "overall_score": 77,
  "overall_comment": "建议复核"
}
```

说明：
1. `overall_score` 沿用当前公式：`T1 + 精筛 + 加分 - 风险扣减`。
2. 评估阈值沿用现有 PASS/REVIEW/REJECT 规则。
3. 前端页面只消费此结构，不再拼接自由文本。

## 4. 采集模块重构方案

涉及：`apps/desktop/src/stores/recruiting.ts`、`stores/recruiting/task-orchestrator.ts`、`stores/recruiting/candidate-import.ts`

核心流程：
1. Sidecar 拉取候选人原始数据。
2. 批内去重（姓名+年龄+地址规则）。
3. 写入 `pending_candidates`（先入待定）。
4. 与 `pending_candidates` / `candidates` 做应用内去重归并。
5. 若开启自动同步：同步到 `candidates`。
6. 同步成功后自动执行 `run_resume_screening`。

调度逻辑：
1. `ONCE`：执行一次后结束。
2. `DAILY`：每天固定时间触发。
3. `MONTHLY`：每月固定日期+时间触发；短月溢出取当月最后一天。

## 5. 候选人模块改造方案

页面：`apps/desktop/src/views/CandidatesView.vue`

改造点：
1. 候选人详情标题行右侧展示当前阶段。
2. 顶部信息区域邮箱下一行展示“候选人简历”。
3. AI 评估区域展示结构化输出（T0/T1/T2/风险/总分/总评）。
4. 按钮区：
- `重新分析`：仅重跑 `run_resume_screening`
- `邀约面试`：带 `candidateId` 跳转面试页
- `遗憾`：直接写 `finalize_hiring_decision(NO_HIRE, reason_code=interview_reject)`
5. 列表升级为后端分页查询：
- 筛选：职位、姓名模糊、阶段
- 排序：默认职位升序 + 评分降序

## 6. 面试模块改造方案

页面：`apps/desktop/src/views/InterviewView.vue`

改造点：
1. 增加待面试人员列表（分页 + 职位/姓名筛选）。
2. 列表按钮：
- `移除面试`
- `生成面试题`
- `已面试`
3. `生成面试题`后打开右侧抽屉，抽屉内承载：
- 候选人信息
- 生成题库
- 题库编辑
- 面后反馈提交
4. 面后反馈录音使用文件上传：
- 先调用 `save_interview_recording`
- 再将返回路径写入 `submit_interview_feedback`
5. `已面试`动作：标记已面试并跳转决策页。

## 7. 决策模块改造方案

页面：`apps/desktop/src/views/DecisionView.vue`

改造点：
1. 增加已面试人员列表（包含历史已决策项）。
2. 分页筛选：职位、候选人姓名、面试是否通过（按人工按钮动作标记）。
3. 列表按钮：
- `查看面试情况`
- `面试通过`（写 `HIRE`, reason_code=`interview_pass`）
- `遗憾`（写 `NO_HIRE`, reason_code=`interview_reject`）
4. 已决策项按钮置灰但支持查看。

## 8. 风险与兼容

1. 兼容风险：保留旧 `listCandidates(stage?)`，新页面逐步迁移到分页 API。
2. 数据迁移风险：`ALTER TABLE` 新字段需兜底默认值，避免旧库启动失败。
3. 调度风险：时区与月末日期计算需统一在后端实现并覆盖测试。
4. 采集风险：不同站点字段不齐，保持“缺失置空，不阻断入待定”。

## 9. 完成标准（DoD）

1. 采集任务可按三种调度模式稳定运行。
2. 采集数据先入 `pending_candidates`，再按配置同步候选人。
3. AI 评估结果结构化可解析，前端完整展示。
4. 候选人/面试/决策页面均支持后端分页筛选。
5. 面试录音上传后可回查本地路径。
6. 快捷决策动作写入 `hiring_decisions` 并同步阶段。
